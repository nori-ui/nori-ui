The library was built with both human and machine readers in mind. Three endpoints make the docs first-class for LLMs and coding agents:

## MCP server

A live [Model Context Protocol](https://modelcontextprotocol.io) server exposing four tools to any MCP-aware client (Claude Desktop, Claude Code, Cursor, Windsurf, Zed):

- `search_components(query)` — fuzzy search by name, description, or tag.
- `get_component_docs(name)` — full docs body for one component.
- `get_component_props(name)` — typed prop list with descriptions and defaults.
- `list_examples(name)` — runnable code snippets keyed by component.

**Server URL:**

```
https://nori-ui.com/mcp
```

### Claude Desktop / Claude Code

Add to your MCP config (`~/.cursor/mcp.json`, `claude_desktop_config.json`, etc.):

```json
{
  "mcpServers": {
    "nori-ui": {
      "url": "https://nori-ui.com/mcp"
    }
  }
}
```

### Cursor

Settings → MCP → Add new server → paste the URL above.

### Any MCP-compatible agent

The endpoint speaks the standard MCP streamable-HTTP transport. Drop the URL into your client's server config — no auth, no install, nothing to configure.

## llms.txt

A flat index of every doc page, optimised for token-efficient discovery. The convention follows [llmstxt.org](https://llmstxt.org).

```
https://nori-ui.com/llms.txt
```

Use this when you want an agent to *find* the right component before fetching its docs.

## llms-full.txt

Every doc page concatenated into a single plain-text file — the entire library in one fetch.

```
https://nori-ui.com/llms-full.txt
```

Use this when context is cheap and you want the agent to have the whole reference up front.

<Callout type="info">
For long-running agents, prefer the MCP server: it's targeted, structured, and avoids token bloat. Use `llms-full.txt` only for one-shot tasks where the model has a large context window to spare.
</Callout>
